设计模式

思考：

  • 我们为什么要设计/抽象，它的本质是什么
  • JavaScript 有什么特点，什么样的设计模式适合这门语言
  • 了解“逻辑”背后的信息比如：满足什么需求？解决什么问题？变化会多频繁？有什么痛点？

（世面上JS设计模式的书基本都在扯淡，很多设计模式是不适合JS的）

模式1：Behavior

  • 在交互需求出发设计一组“动作行为”
  • 在主体对象上应用和扩展这些行为
  • 为什么这么做？

实例1：图片预览（可切换到上一张下一张图片）

分析：Behavior 模式：自定义 preview 行为

思考与练习：

  1. 如果图片要动态加载，怎么修改 setPreviewBehavior?
  2. 如果要兼容旧的浏览器，如IE7，如何处理？
  3. 为上一张、下一张图片切换添加动画

实例2：选取图片，自定义 setSelectionBehavior 行为

模式2：Adapter

  • 针对不同底层实现提供统一的外部接口
  • 让业务层面的代码可以适应不同的底层实现

即针对不同的状况进行判断

Behavior + Adaptor

模式3：Observer（观察者，解耦作用）

Web开发最核心的就是事件驱动，事件是一个异步模型，所以observer（观察者模式）很重要

Observer 用于

  • 同步多个对象的状态变化
  • 解耦各个对象之间的依赖

实例3：同步滚动

模式4：Mediator（中介者）

  • 将命令传递给所有订阅者
  • 与所有订阅者之间维持依赖关系
  • 从而让订阅者彼此之间相互独立

上一个例子，改用 pub/sub

模式5：Composite（复合模式）

  • 对已有组件/库/功能的组合使用
  • 提升代码复用性的最常用手段

实例：命令式连续动画

练习：完善动画例子，增加其他特效动画支持，自己将一些基础组件合成为高级组件

模式6：Partial Application

  • 部分应用/固化参数

实例：resolve path 拼接字符串作为路径，node跨平台，如果直接拼，在windows下会有错误（windows路径用反斜杠）

实例：Logging time 在每个日志上打上时间戳

模式7：Decorate（装饰器）

  • 在原有 API 基础上增加额外的功能

  • 不改变底层库的基础上扩展功能的一种模式

    （不要直接在一些库上修改代码）

实例：让函数只被调用一次

图4

实例：debounce 鼠标停下的时候让小鸟到鼠标位置

如果不增加 debounce，小鸟会卡住

模式8：Proxy（代理模式）

  • 保持原有 API 功能不变，改变执行过程和结果
  • 不改变底层库的基础上获得不同执行结果的手段

实例：拦截异步数据

情景：前端构建完毕，后端服务接口还没有开发好，这时需要调试

图5

实例：mouse 代理 touch 事件

情景：PC端模拟移动端调试

模式9：Aspect（切面）

  • 对代码的执行过程进行监测和修改
  • 不改变代码主体执行流程的基础上控制执行过程

系统/环境默认的一些可控制点

实例：创建切面修改 Swipable

针对封装好的组件没有接口的情况

实例：performance.watch 监控性能

Decorate、Proxy 和 Aspect三者联系：

  • Decorate 强调的是声明和定义
  • Proxy 强调的是功能
  • Aspect 强调的是运行过程

动态语言，可以用 Decorate 来实现 Proxy 和 Aspect

模式10： Lazy load （懒加载）

  • 一些加载好事的资源进行分段、异步加载
  • 前端提升性能，减少响应用户的时间
  • 某些情况下减少网页流量和带宽消耗

实例：懒加载图片
